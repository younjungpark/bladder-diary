# Sync Behavior

## 목적

이 문서는 기록 저장과 동기화의 사용자 관점 계약을 설명한다.

## 핵심 동작

- 기록 추가, 수정, 삭제는 먼저 로컬 DB에 반영된다.
- 원격 반영이 필요한 작업은 sync queue에 적재된다.
- 클라우드 동기화는 기본적으로 꺼져 있으며, 사용자가 명시적으로 켠 뒤에만 원격 업로드, 다운로드, WorkManager 재시도를 수행한다.
- 클라우드 동기화가 켜져 있으면 백그라운드 동기화를 예약하고, 실패하면 WorkManager 재시도로 다시 시도한다.
- 원격 다운로드는 클라우드 동기화가 켜진 사용자 세션 기준으로 가져오며, 로컬 DB에 병합한다.

## 현재 계약

- 날짜 기준은 기기 로컬 타임존 자정이다.
- 삭제는 원격에서 soft delete로 표현된다.
- 클라우드 동기화는 기록 암호화 설정 또는 잠금 해제 후에만 켤 수 있다.
- 동기화가 켜져 있으면 정확한 배뇨 시각, 배뇨량, 절박감, 요실금 여부, 야간뇨 여부, 메모는 `E2EE_RECORD_V1` 기록 본문 암호문 기준으로 동기화된다.
- Supabase에는 날짜, 계정/기록 식별자, 삭제/동기화 메타데이터가 남을 수 있다.
- 동기화가 꺼져 있으면 pending 작업과 로컬 기록은 기기에 남고, 원격 호출이나 백그라운드 동기화 예약은 수행하지 않는다.
- 동기화를 켜면 기존 pending 작업의 백그라운드 업로드를 예약한 뒤, 원격 데이터를 다시 내려받아 병합한다. 이미 동기화 완료된 로컬 기록은 다시 pending으로 바꾸지 않는다. 설정 변경 UI는 네트워크 동기화 완료를 기다리지 않는다.
- 동기화가 켜진 상태에서도 기록 추가, 수정, 삭제는 로컬 저장 완료만 기다린다. 원격 업로드는 백그라운드로 예약되며 입력 UI를 막지 않는다.
- 원격 데이터를 내려받을 때 같은 기록에 로컬 pending/failed 변경이 남아 있으면 원격 데이터가 로컬 변경을 덮어쓰지 않는다.
- sync queue는 현재 로그인 사용자 기준으로 처리하며, 동기화를 다시 켤 때 이전 오류 문구는 다음 재시도 결과가 나올 때까지 초기화한다.
- 동기화를 끄면 이후 업로드와 다운로드를 멈추고 예약된 WorkManager 동기화를 취소한다. 이미 클라우드에 저장된 데이터는 자동으로 삭제하지 않는다.
- 사용자가 동기화를 켠 상태에서 온라인으로 복귀하면 pending 작업이 다시 업로드될 수 있다.
- 기존 평문 또는 메모 전용 E2EE 클라우드 기록은 사용자가 E2EE를 설정하거나 잠금 해제한 뒤 내려받아 로컬에 병합하고, 같은 기록 ID로 `E2EE_RECORD_V1` 암호문을 재업로드한다.
- 현재 Supabase 테이블의 평문 민감 필드는 재업로드 성공 후 비워지지만, 과거 백업이나 로그에 남은 평문까지 앱이 즉시 삭제한다고 보장할 수는 없다.
- 회원탈퇴는 운영자 확인용 탈퇴 요청을 먼저 원격에 기록하고, 원격 앱 데이터를 삭제한 뒤 로컬 DB와 기기 보안 상태를 초기화하고 로그아웃한다.

## 실패 시 기대 동작

- 동기화 실패가 있더라도 로컬 데이터는 즉시 사라지지 않는다.
- pending / failed 상태는 UI와 저장소 흐름에서 확인 가능해야 한다.
- 인증 토큰 만료가 발생하면 세션 갱신 후 재시도를 시도한다.
- 회원탈퇴 요청 기록이나 원격 삭제가 실패하면 로컬 데이터는 유지하고 사용자에게 오류 메시지를 표시한다.
