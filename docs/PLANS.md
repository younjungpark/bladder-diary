# PLANS

## 목적

실행 계획을 저장소 안에서 버전 관리하기 위한 안내 문서이다.
에이전트가 긴 작업을 할 때 대화 히스토리에만 의존하지 않도록 계획을 파일로 남긴다.

이 문서는 저장소 내부 실행 메모 운영 규칙을 다루며,
이슈 트래커나 커밋 히스토리 자체를 대체하지 않는다.

## 상태 관리 원칙

- 외부 이슈 트래커를 쓰고 있다면 그 상태를 source of truth로 둔다.
- `exec-plans`는 복잡한 작업의 실행 메모를 저장하는 보조 수단으로 사용한다.
- 완료된 작업을 모두 저장소에 보관하지 않는다.
- 코드와 기준 문서에 결과가 충분히 반영된 작은 작업은 완료 후 계획 문서를 삭제한다.
- 완료 후에도 장기 참조 가치가 남는 정책, 큰 리팩터링, 마이그레이션 기록만 `completed/`에 남긴다.

## 기본 진입 규칙

- 작은 수정이나 단일 파일 작업에서는 이 문서를 기본으로 읽지 않는다.
- 여러 파일, 여러 계층, 계약 변경, 장기 작업일 때만 이 문서를 연다.
- 활성 계획은 현재 작업과 직접 관련된 문서 1개만 연다.
- 완료 계획은 기본 진입 문서가 아니다. 과거 결정 추적이 필요할 때만 연다.

## 위치

- 진행 중 계획: `exec-plans/active/`
- 완료된 계획: `exec-plans/completed/`
- 횡단 기술 부채 추적: `exec-plans/tech-debt-tracker.md`

## 언제 계획을 만들까

- 변경 범위가 여러 파일 또는 여러 계층에 걸칠 때
- 동기화, 보안, 데이터 모델, 사용자 흐름이 함께 바뀔 때
- 테스트, 문서, 운영성 갱신이 함께 필요한 작업일 때
- 작업이 하루 이상 이어질 가능성이 있을 때

작은 수정이나 반나절 안에 끝나는 단일 목적 작업은
계획 문서를 만들지 않아도 된다.

## 계획 문서 권장 항목

- 빠른 요약
- 배경
- 목표와 비목표
- 현재 상태
- 작업 단계
- 검증 방법
- 결정 로그
- 후속 작업

## 작업 종료 시 정리 규칙

1. 구현, 테스트, 관련 기준 문서 갱신을 먼저 끝낸다.
2. 외부 이슈 트래커나 TODO를 쓰고 있다면 상태를 정리한다.
3. 아래 중 하나를 선택한다.
   - 장기 참조 가치가 있으면 `completed/`로 이동한다.
   - 장기 가치가 없으면 계획 문서를 삭제한다.

## 커밋 메시지 규칙

- 커밋 제목은 한글로 작성한다.
- 커밋 본문도 한글로 작성한다.
- 제목은 변경 의도를 짧게 요약하고, 본문에는 배경과 영향 범위를 덧붙인다.

## 현재 활성 계획

- 현재 별도 활성 계획 없음
