# 배뇨일지 (BladderDiary) 안드로이드 앱 출시 준비 체크리스트

이 문서는 배뇨일지(BladderDiary) 앱을 구글 플레이스토어(Google Play Store)에 출시하기 위해 사전에 점검하고 준비해야 할 필수 항목들을 정리한 가이드입니다.

---

## 1. 앱 스토어 에셋 (Assets) 준비
스토어 등록 시 사용자들이 가장 먼저 보게 되는 필수 이미지 및 텍스트 항목입니다.

- [ ] **고해상도 아이콘 (App Icon)**
  - 크기: 512 x 512 픽셀
  - 형식: 32비트 PNG (투명한 배경 허용됨)
- [ ] **스토어 그래픽 (Feature Graphic)**
  - 크기: 1024 x 500 픽셀
  - 형식: JPEG 또는 24비트 PNG (알파 채널 없음)
  - 앱의 대표적인 기능이나 브랜드를 한눈에 보여주는 배너 이미지
- [ ] **스크린샷 (Screenshots)**
  - 최소 2장 필수 (권장: 4~5장)
  - 주요 화면 캡처: 메인 캘린더 화면, 배뇨 타이머/등록 화면, 통계 화면, PIN 잠금 설정 화면 등
- [ ] **앱 제목 (Title)**
  - 30자 이내 (예: `배뇨일지 - 쉽고 깔끔한 배뇨 기록 및 관리`)
- [ ] **간단한 설명 (Short Description)**
  - 80자 이내의 앱 요약 설명
- [ ] **자세한 설명 (Full Description)**
  - 최고 4000자. 앱의 핵심 기능, 장점, 활용 방법 등을 상세히 기재 (예: 백업/동기화 지원, 핀 잠금 보안 등)

---

## 2. 정책 및 법적 필수 요건 (Privacy & Policy)
건강/의료 관련 데이터를 다루고(배뇨 기록), 자체 보안 기능(PIN)을 포함하고 있으므로 정책 관련 준비가 중요합니다.

- [ ] **개인정보 처리 방침 (Privacy Policy)**
  - **필수 항목**: 앱이 어떤 데이터를 수집하고, 어떻게 보호하며, 어디(서버)에 저장하는지 명시한 문서 URL이 필요합니다.
  - *Tip*: Notion, GitHub Pages, Google Sites 등을 활용하여 무료로 호스팅할 수 있습니다.
- [ ] **데이터 보안 양식 작성 (Data Safety Section)**
  - 플레이 콘솔 내에서 작성. (예: `개인정보`, `비밀번호(로그인)`, `건강 데이터` 등의 수집 여부 및 암호화 전송 여부 체크)

---

## 3. 앱 내부 점검 및 마무리 개발 (App Polish)
실제 릴리스(Release) 빌드를 뽑기 전 최종적으로 코드와 빌드 환경을 다듬는 과정입니다.

- [ ] **앱 런처 아이콘(Launcher Icon) 적용**
  - 안드로이드 기기 화면에 노출될 아이콘 (Adaptive Icon 등 적용 여부 점검)
- [ ] **스플래시 스크린(Splash Screen) 확인**
  - 앱 로딩 시 진입하는 첫 화면 브랜딩 일관성 확인
- [ ] **앱 버전 업데이트 (`build.gradle`)**
  - `versionCode` : 1 (정수, 업데이트 시마다 +1 증가시켜야 함)
  - `versionName` : "1.0.0" (사용자에게 노출되는 버전)
- [ ] **출시용 키보안 (App Signing Key) 생성**
  - Android Studio를 통해 `.jks` 또는 `.keystore` 파일을 생성합니다.
  - *경고*: **이 키 파일과 비밀번호는 절대 분실하면 안 됩니다.** (이후 앱 업데이트 시 반드시 필요)
- [ ] **삼성 브라우저 등 서드파티 브라우저 소셜 로그인 이슈 해결**
  - 증상: 크롬(Chrome)이 아닌 삼성 기본 브라우저 설정 시 OAuth / 소셜 로그인 창에서 앱으로 리다이렉트(복귀)되지 않는 문제. 
  - 해결 방법: AndroidManifest.xml의 `intent-filter`에 설정한 `android:scheme` 및 `data` 구성이 서드파티 브라우저 환경에서도 제대로 호출되도록 App Link/Deep Link 구현체를 점검하고, 필요한 경우 Custom Tabs Intent 방식으로 강제하거나 보완 로직(`browser fallback`)을 추가합니다.
- [ ] **Release 프로가드(R8) 규칙 적용 및 테스트**
  - `minifyEnabled true` 상태에서 크래시가 발생하지 않는지, 서버 통신(Retrofit/Moshi 등) 시 난독화로 인한 직렬화 오류가 없는지 `Release` 변형으로 한 번 테스트 빌드하여 확인합니다.

---

## 4. 최종 빌드 및 스토어 제출
- [ ] **AAB (Android App Bundle) 생성**
  - 구글 플레이스토어에서는 이제 APK 대신 `.aab` 확장자의 앱 번들 파일 제출을 요구합니다.
  - Android Studio: `Build` -> `Generate Signed Bundle / APK...` -> `Android App Bundle` 선택
- [ ] **구글 플레이 콘솔 (Google Play Console) 가입 및 등록**
  - 개발자 계정 등록 (평생 1회, $25 결제 필요)
  - 새 앱 만들기 생성 후, 프로덕션(Production) 트랙이나 내부 테스트(Internal Test) 트랙에 AAB 업로드
- [ ] **심사(Review) 대기**
  - 모든 양식(콘텐츠 등급, 타겟 연령층 등)을 채운 후 최초 제출을 하면, 구글의 앱 심사가 진행됩니다. (통상 3일 ~ 일주일 소요)

> **💡 첫 출시 추천 전략 (MVP 출시)**
처음부터 완벽하게 런칭하기보다는, **"내부 테스트"** 또는 **"공개 테스트(Open Beta)"** 트랙에 먼저 앱을 등록하여 본인과 주변 지인들 폰에 직접 다운로드 받아보는 것을 강력히 권장합니다. 이후 발견된 버그를 고치고 프로덕션(정식 출시)으로 전환하는 것이 안전합니다.
